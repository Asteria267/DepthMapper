## тЬж What This Is

Real depth sensors тАФ like the ones in self-driving cars, iPhone LiDAR, or industrial robots тАФ measure distance by shooting photons at the world and timing how long they bounce back. That's Time-of-Flight. That's LiDAR. That's structured light.

DepthMapper does something wilder: it looks at a single 2D RGB frame from your webcam and **predicts the distance of every pixel** using a neural network trained on millions of real depth scans. No infrared. No laser. No stereo rig. Just math and a lot of learned intuition about how the world looks.

This is called **monocular depth estimation**, and MiDaS is one of the best open-source models for it.

The output: a live colorized **depth heatmap** where warm colors (yellow/orange) are close and dark colors (purple/black) are far тАФ updated in real time as you move in front of your camera.

## тЬж Hardware Concept

This project is the **software equivalent of a physical depth sensor**:

| Real Hardware | What It Does | DepthMapper Equivalent |
|---|---|---|
| Intel RealSense D435 | Structured light depth camera | `MiDaSBackend.estimate()` |
| iPhone LiDAR Scanner | dToF depth sensing | `ONNXBackend.estimate()` |
| Velodyne VLP-16 | Spinning laser LiDAR | `export_point_cloud()` CSV |
| SICK LMS | Time-of-flight rangefinder | Center-region depth reading |

The abstraction is intentional. In v2.0, you replace `backend.estimate(frame)` with a RealSense SDK call and the **entire pipeline stays identical** тАФ colorizer, histogram, CSV exporter, dashboard. That's the point.

## тЬж Output: Point Cloud CSV

Pressing SPACE exports a file like this:

```
x,y,depth_norm,depth_mm
0,0,0.8731,6984
4,0,0.8612,6889
8,0,0.7943,6354
...
```

- `x`, `y` тАФ pixel coordinates (every 4th pixel, ~19,200 points at 640├Ч480)
- `depth_norm` тАФ normalized depth [0.0 = near, 1.0 = far]
- `depth_mm` тАФ rough metric estimate (mapped 0тАУ8000mm as demo scale)

This is the same format used in real LiDAR pipelines. In a production system you'd add RGB columns and calibrate the metric scale against a known reference object.

## тЬж Common Fixes

**Depth map is all one color**
тЖТ Normalization wasn't applied. Already fixed тАФ `normalize_depth()` stretches values to 0тАУ255 before colorizing.

**torch download is huge / slow**
тЖТ The script uses `MiDaS_small` (~80MB), not the full model. Or skip torch entirely and use ONNX Runtime instead (~25MB).

**torch.hub fails / no internet**
тЖТ ONNX Runtime fallback activates automatically. No action needed.

**Slow FPS on CPU**
тЖТ 2тАУ5 FPS is expected for MiDaS-small on CPU. This is normal. On Apple M-series chips it uses MPS (Metal GPU) and runs noticeably faster.

**CSV is empty or wrong shape**
тЖТ Fixed тАФ the vectorized `np.mgrid` exporter correctly maps pixel coordinates to depth values.

**Webcam not found**
тЖТ Script tries index 0 then 1. Check that your camera isn't in use by another app.

## тЬж v2.0 Hardware Swap

When you're ready to go from software simulation to real hardware, the swap is one function:

```python
# Replace this:
depth_map = backend.estimate(frame)

# With this (Intel RealSense D435):
import pyrealsense2 as rs
pipe = rs.pipeline()
pipe.start()
frames = pipe.wait_for_frames()
depth_frame = frames.get_depth_frame()
depth_map = np.asarray(depth_frame.get_data(), dtype=np.float32)
```

Everything else тАФ the dashboard, histogram, CSV exporter, colorizer тАФ stays exactly the same.
